DAMM · DDI CONFIDENCIAL · PROPUESTA TÉCNICA Logística HORECA · 2026

Modelo Matemático
VRP + Optimización de Carga

Sistema conjunto de decisión operacional: Vehicle Routing Problem + Bin Packing por Palés.
Aplicado a flota real DDI: 11 camiones grandes (6 palés), 4 camiones medianos (8 palés), 1 furgoneta (3 palés).
01 Principio fundamental

La hipótesis central del sistema es que ruta y carga NO pueden optimizarse por separado. En distribución HORECA con acceso lateral, la configuración de palés determina la accesibilidad de cada parada, y el orden de paradas determina qué palé debe estar disponible en cada momento. Ambas dimensiones se retroalimentan mutuamente.

Objetivo: calcular simultáneamente la secuencia óptima de clientes y la distribución física de palés, minimizando el coste operativo total: tiempo de ruta + tiempo de descarga + fricción operativa + movimientos innecesarios.

03 Función objetivo

El sistema minimiza el coste operativo total, combinando cuatro componentes con pesos calibrables:

/* Función objetivo principal */ min Z = α·T_ruta + β·Σ T_desc_i + γ·Σ fric_i + δ·Interference donde: T_ruta = Σ d_ij · x_ij /* distancia/tiempo de ruta */ T_desc_i = base_i + β₁·fric_i + β₂·barrel_i /* tiempo descarga por parada */ fric_i = Σ_k [ p_ik · (N - pos_k) · overload_k ] /* fricción por posición */ Interference = Σ_ij [ p_ik · p_jk · conflict_ij ] /* conflicto entre mercancías */ Pesos base DDI (calibrables): α = 1.0 β = 1.2 γ = 0.8 δ = 0.6

T_ruta

  • Tiempo de desplazamiento entre paradas
  • Incluye ventanas horarias de clientes
  • Penaliza rutas con backtracking

T_desc_i (fricción)

  • Función de posición del palé
  • Barriles: penalización +20 sin acceso rápido
  • Clientes >40 uds sin palé exclusivo: +10

Interference

  • Conflicto entre retornables y carga nueva
  • Incompatibilidad física de SKUs en palé
  • Clientes de distinto perfil en mismo palé
04 Restricciones del sistema

Restricciones VRP

  • Cada cliente visitado exactamente una vez
  • Ventanas horarias respetadas (hard/soft según cliente)
  • Capacidad de carga del vehículo no superada
  • Regreso al depósito al final de jornada
  • Número máximo de paradas por ruta

Restricciones de palés

  • Cada cliente asignado a exactamente un palé: Σ_k p_ik = 1
  • Capacidad por palé: Σ_i p_ik · vol_i ≤ cap_k
  • Barriles solo en palés de acceso rápido (pos_k ≥ N-1)
  • Máx. 2 barriles apilados por palé
  • Palé fondo reserva 20% para retornables

Restricciones físicas

  • Acceso lateral: palés ordenados linealmente (1=fondo, N=puerta)
  • Clientes ≥50 uds: palé exclusivo obligatorio si hay espacio
  • Retornables vacíos: separados físicamente de carga nueva
  • Barriles: zona lateral izquierda, siempre accesibles

Restricciones operativas

  • Secuencia de descarga = inverso de orden de carga
  • Primer palé accesible (puerta) = primeras paradas de ruta
  • Clientes con barril deben estar en los primeros N/3 de ruta
  • Máx. interferencia tolerable por palé: umbral configurable
05 Arquitectura algorítmica

El sistema usa una arquitectura heurística en dos fases acopladas, industrializable con tiempos de respuesta <30 segundos para rutas de hasta 30 paradas:

FASE 1
Construcción inicial
Clarke-Wright savings + nearest neighbor. Genera solución factible de ruta y asignación inicial de palés.
FASE 2
Mejora conjunta
2-opt/3-opt sobre rutas + bin-reassignment sobre palés. Iteración hasta convergencia o límite de tiempo.
FASE 3
Ajuste por reglas
Motor de reglas de negocio: barriles, retornables, clientes VIP, ventanas horarias.
FASE 4
Output accionable
Generación de recomendación con explicabilidad: orden, palés, zonas, KPIs.
/* Pseudocódigo del motor de decisión */ function OptimizeRoute(orders, fleet, constraints): route ← ClarkeWright(orders) // construcción inicial VRP pallets ← InitialPacking(route, fleet) // bin packing por capacidad for iter in 1..MAX_ITER: route ← TwoOpt(route, pallets) // mejora ruta considerando carga pallets ← ReassignPallets(route) // reasigna palés con nueva ruta if ΔZ < ε: break // convergencia pallets ← ApplyBusinessRules(pallets) // barriles, retornables, VIPs return Explain(route, pallets) // output con justificación
06 Motor de decisión híbrido

La asignación de cada parada al modo de carga óptimo se realiza mediante un clasificador basado en reglas y scoring, no puramente SKU-based ni cliente-based:

/* Score de modo de carga para cada parada i */ score_sku(i) = w1·(uds_i ≤ 5) + w2·(skus_i ≤ 2) + w3·(¬barrel_i) score_hybrid(i) = w1·(6≤uds_i≤50) + w2·(3≤skus_i≤10) + w3·(urban_zone_i) score_client(i) = w1·(uds_i > 50) + w2·(skus_i > 10) + w3·(barrel_i) mode(i) = argmax { score_sku, score_hybrid, score_client } /* Pesos calibrados con datos Mollet 11/02/2026 */ w1 = 0.40 w2 = 0.35 w3 = 0.25

Modo SKU-based

  • ≤5 uds, 1-2 SKUs únicos
  • Sin barril, zona densa
  • Agrupa con clientes similares
  • Palé compartido por tipo de producto

Modo Híbrido ★

  • 6–50 uds, 3–10 SKUs
  • Posible barril, ruta urbana
  • Sub-zona propia en palé compartido
  • Accesibilidad garantizada sin palé exclusivo

Modo Cliente-aware

  • >50 uds, >10 SKUs + barril
  • Acceso complicado o cliente VIP
  • Palé exclusivo obligatorio
  • Posición near-door en camión
07 Integración de flota real DDI

El modelo incorpora las capacidades reales de los vehículos DDI para calcular factibilidad y asignar rutas a vehículos de forma óptima:

/* Capacidades por tipo de vehículo */ FLEET = { "camion_grande": { count: 11, pallets: 6, cap_per_pallet: 120, total_cap: 720 }, "camion_mediano": { count: 4, pallets: 8, cap_per_pallet: 90, total_cap: 720 }, "furgoneta": { count: 1, pallets: 3, cap_per_pallet: 60, total_cap: 180 }, } /* Asignación ruta → vehículo */ assign_vehicle(route) = if total_units(route) > 500 or barrels(route) > 5: camion_mediano elif total_units(route) > 150: camion_grande else: furgoneta
Nota de calibración: los umbrales de asignación deben validarse con datos históricos de DDI. El modelo permite ajustar los parámetros por zona geográfica (urbana vs periurbana), tipo de cliente (HORECA premium vs volumen) y época del año (temporada alta cerveza).

09 Explicabilidad del sistema

Cada decisión del modelo debe ser justificable para ser adoptada por el equipo operativo. El sistema genera automáticamente las razones de cada asignación:

¿Por qué esta ruta?

  • Minimiza backtracking en zona Mollet Norte
  • Clientes con barril agrupados al inicio (acceso rápido)
  • Ventanas horarias de FRANKFURT respetadas (08:00–10:00)
  • Secuencia optimiza retorno retornables al depósito

¿Por qué esta organización de palés?

  • FRANKFURT en palé 6 (puerta): 70 uds → descarga sin reorganización
  • Barriles siempre en acceso lateral izquierdo
  • Clientes pequeños agrupados en palés fondo: reducen movimientos
  • Retornables reservados en palé 1: evita mezcla con carga nueva

Trade-offs optimizados

  • +28 min preparación almacén → −72 min descarga en ruta
  • Palé exclusivo para FRANKFURT → +0.5 palé de capacidad perdida
  • Retornables en fondo → conductor los recoge sin descargar carga nueva

Retroalimentación ruta ↔ carga

  • Si FRANKFURT se adelanta en ruta → palé 6 se mantiene accesible más tiempo
  • Cambio de vehículo (6→8 palés) → DR0017 distribuye barriles en 2 palés
  • Nuevo cliente urgente → sistema recalcula posición en palé sin modificar ruta
10 Hoja de ruta de industrialización

Fase I — Validación (4–6 semanas)

  • Calibrar pesos α,β,γ,δ con datos históricos DDI
  • Validar capacidades reales por palé y vehículo
  • Pilotar con 2 rutas reales en Mollet
  • Medir ahorro real vs estimado del modelo

Fase II — Motor automático (8–12 semanas)

  • API de optimización: input pedidos → output palés+ruta
  • Integración con sistema de pedidos DDI
  • Dashboard de control para jefes de ruta
  • Generación de hoja de carga por conductor

Fase III — Escala (3–6 meses)

  • Extensión a otras zonas geográficas DDI
  • Incorporación de ML para predicción de fricción
  • Optimización multi-día y multi-ruta simultánea
  • Retroalimentación en tiempo real (GPS + incidencias)

KPIs objetivo del sistema

  • −15 a −25% tiempo descarga por jornada
  • −30 a −50% movimientos internos evitados
  • +8 a +12% coste preparación almacén (neto positivo)
  • >80% paradas con acceso directo sin reorganización
Conclusión: el modelo matemático propuesto es industrializable, explicable y directamente aplicable a la operativa real de Damm/DDI. La arquitectura heurística garantiza tiempos de respuesta operativos (<30 s) y la integración explícita de la flota real (6/8/3 palés) convierte el output en una recomendación accionable por cualquier conductor, sin requerir conocimiento técnico del modelo subyacente.
